対象ページ: 従業員名簿ページ

このドキュメントは、「従業員名簿ページ」のフロントエンドとバックエンドを連携させるためのAPI仕様を定義します。

これベースで開発ヨロ！

0. DB設計

従業員一覧は以下のテーブルをベースにしています。

roles（権限マスタ）

CREATE TABLE roles (
    role_id SERIAL PRIMARY KEY,            -- 自動採番
    role_name VARCHAR(50) UNIQUE NOT NULL,  -- 権限名(例:"管理者"や"一般")
    is_admin BOOLEAN NOT NULL DEFAULT FALSE --追加 09/25
);

production_lines（ラインマスタ）

CREATE TABLE production_lines (
    line_id SERIAL PRIMARY KEY,
    line_name VARCHAR(100) UNIQUE NOT NULL
);

employees（従業員）

CREATE TABLE employees (
    employee_id SERIAL PRIMARY KEY,
    employee_name VARCHAR(100) NOT NULL,
    employee_user_id VARCHAR(50) UNIQUE NOT NULL,
    employee_password VARCHAR(255) NOT NULL,
    employee_is_active BOOLEAN DEFAULT TRUE,
    employee_role_name VARCHAR(50) NOT NULL,
    employee_line_name VARCHAR(50) NOT NULL,
    employee_special_notes TEXT,
    employee_color_code CHAR(6)
);

検索パラメータ（name_like, line_name, is_active）は employees の列を対象に絞り込みます。

1. 従業員一覧取得

GET /api/employees 優先度: S

従業員のリストをまとめて取得するAPIです。検索フィルターにも対応します。

クエリパラメータ (絞り込み検索)

パラメータ 説明 例
name_like 氏名での部分一致検索。 ?name_like=山田
line_name 担当ラインのIDで絞り込み。 ?line_name=組立
is_active 在籍状況で絞り込み (true: 在籍中, false: 退職済)。 ?is_active=true

2. 特定の従業員取得

GET /api/employees/{id} 優先度: S

ID を指定して、特定の従業員1人の詳細情報を取得します。

成功レスポンス (200 OK)

{
    "employee_id": 1,
    "employee_name": "山田 太郎",
    "employee_user_id": "t.yamada",
    "is_active": true,
    "role_name": "リーダー",
    "line_name": "組立",
    "special_notes": "フォークリフト免許保持",
    "color_code": "a3a8e6"
}

エラーレスポンス (404 Not Found)

{
    "error": "Employee not found"
}

3. 従業員追加

POST /api/employees 優先度: S

新しい従業員を登録するためのAPIです。データは JSON 形式で送信します。

リクエストボディ

{
    "employee_name": "鈴木 次郎",
    "employee_user_id": "j.suzuki",
    "password": "secure_password_123",
    "role_id": 2,
    "line_id": 1,
    "color_code": "a3e6c8",
    "special_notes": "新人"
}

成功レスポンス (201 Created)

登録が成功すると、ID が付与された従業員データが返却されます。

{
    "employee_id": 3,
    "employee_name": "鈴木 次郎",
    "employee_user_id": "j.suzuki",
    "is_active": true,
    "role_id": 2,
    "line_id": 1,
    "special_notes": "新人",
    "color_code": "a3e6c8"
}

4. 従業員情報更新

PUT /api/employees/{id} 優先度: A

IDで指定した従業員の情報を全件更新します。部分更新は PATCH が適切ですが、本設計では単純化のため PUT を利用します。

リクエストボディ

{
    "employee_name": "鈴木 次郎",
    "employee_user_id": "j.suzuki",
    "password": null,
    "role_id": 1,
    "line_id": 2,
    "is_active": true,
    "color_code": "a3e6c8",
    "special_notes": "検査のプロ"
}

成功レスポンス (200 OK)

{
    "employee_id": 3,
    "employee_name": "鈴木 次郎",
    "employee_user_id": "j.suzuki",
    "is_active": true,
    "role_id": 1,
    "line_id": 2,
    "special_notes": "検査のプロ",
    "color_code": "a3e6c8"
}

5. 従業員削除

DELETE /api/employees/{id} 優先度: S

IDで指定した従業員を削除します。実際には `is_active` を false にする論理削除の方が安全ですが、本設計では暫定的に物理削除を前提としています。

成功レスポンス (204 No Content)

削除が成功した場合、レスポンスボディは空となります。
