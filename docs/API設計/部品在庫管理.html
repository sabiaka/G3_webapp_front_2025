部品在庫管理API 設計書

← 管理画面に戻る

対象ページ: 部品在庫管理ページ

このドキュメントは、「部品在庫管理ページ」の機能を実装するためのAPI仕様を定義します。箱IDは廃止しています。

1. DB設計

DB 設計は以下のとおりです。シンプルな構成としています。

racks (棚) テーブル

CREATE TABLE racks (
    -- 棚のID、自動で番号が振られる
    rack_id SERIAL PRIMARY KEY,
    -- 棚の名前 (例: スプリング・小物資材ラック)
    rack_name VARCHAR(255) NOT NULL UNIQUE,
    -- 棚の行数 (A, B, C...)
    rows INT NOT NULL CHECK (rows > 0),
    -- 棚の列数 (1, 2, 3...)
    cols INT NOT NULL CHECK (cols > 0)
);

slots (スロット/箱) テーブル

CREATE TABLE slots (
    -- スロットのID、これも自動
    slot_id SERIAL PRIMARY KEY,
    -- どの棚に属してるかのID
    rack_id INT NOT NULL,
    -- 場所を示す識別子 (例: 'A-1', 'C-4')
    slot_identifier VARCHAR(10) NOT NULL,

    -- ↓部品が入ってるときだけ値が入る↓
    part_name VARCHAR(255),
    part_model_number VARCHAR(100),
    quantity INT CHECK (quantity >= 0),
    color_code VARCHAR(7),

    -- 同じ棚の同じ場所に2個は置けないようにする設定
    UNIQUE (rack_id, slot_identifier),
    -- 存在しない棚には部品を置けないようにする設定
    FOREIGN KEY (rack_id) REFERENCES racks(rack_id) ON DELETE CASCADE
);

2. 棚 (Rack) API

2.1. 棚一覧の取得

GET /api/racks

登録されている棚の一覧を取得するAPIです。左側のラック一覧を表示する際に利用します。

成功レスポンス (200 OK)

[
    {
        "rack_id": 1,
        "rack_name": "スプリング・小物資材ラック",
        "rows": 3,
        "cols": 4
    },
    {
        "rack_id": 2,
        "rack_name": "ウレタンフォーム保管棚",
        "rows": 3,
        "cols": 6
    }
]

2.2. 特定の棚の詳細取得

GET /api/racks/{rack_id}

指定した棚の詳細情報を取得します。内部に格納されている部品（スロット）の情報もまとめて取得できます。

成功レスポンス (200 OK)

{
    "rack_id": 1,
    "rack_name": "スプリング・小物資材ラック",
    "rows": 3,
    "cols": 4,
    "slots": {
        "A-1": { "part_name": "ポケットコイル (シングル)", "part_model_number": "PC-S-H20", "quantity": 25, "color_code": "4A90E2" },
        "A-2": { "part_name": "ポケットコイル (シングル)", "part_model_number": "PC-S-H20", "quantity": 25, "color_code": "4A90E2" },
        "A-3": null,
        "A-4": { "part_name": "ボンネルコイル (セミダブル)", "part_model_number": "BC-SD-H18", "quantity": 20, "color_code": "50E3C2" },
        "B-1": { "part_name": "コーナーガード", "part_model_number": "CG-STD", "quantity": 500, "color_code": "9B9B9B" }
        // ...以下続く
    }
}

2.3. 新規ラックの作成

POST /api/racks

新しい棚（ラック）を追加するときのAPIです。名前とサイズを指定して POST します。

リクエストボディ

{
    "rack_name": "生地・充填材ラック",
    "rows": 2,
    "cols": 3
}

成功レスポンス (201 Created)

{
    "rack_id": 3,
    "rack_name": "生地・充填材ラック",
    "rows": 2,
    "cols": 3,
    "message": "新しいラックが作成されました。"
}

2.4. ラックの削除

DELETE /api/racks/{rack_id}

不要になった棚を削除するAPIです。URL で ID を指定して実行します。

成功レスポンス (200 OK)

{
    "message": "ラック「スプリング・小物資材ラック」を削除しました。"
}

3. 箱 (Box/Part) API

3.1. 新規部品の格納

POST /api/racks/{rack_id}/slots/{slot_identifier}

空のスロットに新しい箱を格納するときに利用します。`slot_identifier` は `A-1` のような形式です。

リクエストボディ

{
    "part_name": "ラテックスフォーム 40mm (D)",
    "part_model_number": "LTX-T40-D",
    "quantity": 20,
    "color_code": "F8E71C"
}

成功レスポンス (201 Created)

{
    "message": "A-1に部品を格納しました。",
    "slot": {
        "slot_identifier": "A-1",
        "part_name": "ラテックスフォーム 40mm (D)",
        "part_model_number": "LTX-T40-D",
        "quantity": 20,
        "color_code": "F8E71C"
    }
}

3.2. 部品情報の編集

PUT /api/racks/{rack_id}/slots/{slot_identifier}

既存箱の情報を編集するときは PUT を使用します。数量変更やカラー変更などに対応します。

リクエストボディ

{
    "part_name": "【改】ラテックスフォーム 40mm (D)",
    "part_model_number": "LTX-T40-D-rev2",
    "quantity": 25,
    "color_code": "4A90E2"
}

成功レスポンス (200 OK)

{
    "message": "A-1の部品情報を更新しました。",
    "slot": {
        "slot_identifier": "A-1",
        "part_name": "【改】ラテックスフォーム 40mm (D)",
        "part_model_number": "LTX-T40-D-rev2",
        "quantity": 25,
        "color_code": "4A90E2"
    }
}

3.3. 部品の削除

DELETE /api/racks/{rack_id}/slots/{slot_identifier}

棚から箱を削除する（＝スロットを空にする）ときに使うAPI。

成功レスポンス (200 OK)

{
    "message": "A-1の部品を削除しました。"
}

3.4. 部品の移動

POST /api/racks/move/

箱を別の場所に移動するAPI。移動元と移動先の情報を送る！

リクエストボディ

{
    "from_rack_id": 1,
    "from_slot_identifier": "A-1",
    "to_rack_id": 1,
    "to_slot_identifier": "C-3"
}

成功レスポンス (200 OK)

{
    "message": "部品を A-1 から C-3 へ移動しました。"
}

{rack_id}では、データのやり取りはしない。全てリクエストJSONで完結させるものとする。

3.5. 部品の使用

POST /api/racks/{rack_id}/slots/{slot_identifier}/use

部品を使用した際に在庫数を減らすためのAPIです。「何個使用したか」をサーバーへ伝えます。

リクエストボディ

{
    "quantity_to_use": 5
}

成功レスポンス (200 OK)

{
    "message": "部品を5個使用しました。",
    "remaining_quantity": 20
}

使用後の在庫が 0 個になった場合、サーバー側で自動的にそのスロットを空状態として扱います。

4. 棚QRコード仕様

QR コードは「棚 QR」の 1 種類とし、入庫と出庫の双方に対応します。

棚QRコードのデータ構造

棚の各場所に貼る QR コードは、スキャンすると以下の JSON 文字列を取得できるようにします。

{
    "type": "rack_slot",
    "rack_id": 1,
    "slot_identifier": "A-1"
}

【補足】QR入庫の流れ

  • 1. モーダルで新しい部品の情報を入力します。
  • 2. カメラで配置したい場所の「棚 QR コード」をスキャンします。
  • 3. スキャン結果から棚の位置（例: `rack_id: 1`, `slot_identifier: 'A-1'`）を取得します。
  • 4. API (`POST /api/racks/1/slots/A-1`) を呼び出し、入力した部品情報をサーバーへ送信して格納を完了します。

【補足】QR出庫の流れ

  • 1. FAB メニューから「QR 出庫」を選択します。
  • 2. カメラで出庫対象の棚に貼付された「棚 QR コード」をスキャンします。
  • 3. スキャン結果を基にアプリが対象スロットの部品情報を表示します。
  • 4. 「部品を使用」モーダルで使用した数量を入力します。
  • 5. API (`POST /api/racks/1/slots/A-1/use`) を呼び出し、サーバー側の在庫数を更新して完了します。
