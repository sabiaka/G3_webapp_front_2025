対象ページ: 画像検査ステータス

このドキュメントは、「画像検査ステータス」ページの最新フロント実装に合わせたAPI/DB設計を定義します。WebSocket は廃止し、軽量な GET ポーリングで変更検知→ フロントは必要なデータを GET で再取得する方式に変更しました。

改訂概要

  • リアルタイム更新: WebSocket → 軽量GETポーリングに変更（サーバは直近イベントID/名前を公開。フロントは数秒おきに比較し、変化時のみデータAPIを再取得）。
  • DBスキーマ: 単一テーブルから ロット＋カメラショット の2層構造へ再設計。1ロット内の同一カメラの複数ショットを保存可能。
  • API整備: クエリは section と date を基本に統一。summary と lots を中核に、任意で lot-shots や dates を追加。
  • ショット詳細: 4K / FHD ごとにエンドポイントを分割し、4k_seq（DB 列: c4k_seq）・fhd_seq で新たな撮影シーケンスを識別。
  • ステータス表記: サーバは PASS/FAIL を返却。フロントの UI 表示（OK/NG 等）はフロント側でマッピング。
  • 画像命名: {lot_id}_{camera_id}-{seq}.jpg を推奨（例: MAT-10001_B-spring01-02.jpg）。

カメラ構成と命名規則

検査システムは複数のカメラで構成されます。各カメラには、その物理的な位置と役割を示す一意のIDが割り当てられます。カメラIDは通常、{層}-{役割}-{番号} という形式で命名されます。

1. データベーススキーマ（刷新）

PostgreSQLを想定。1ロット内で同一カメラの複数ショットを保持し、ロット単位の集計を容易にします。


-- ロット（製品ID等に相当）
CREATE TABLE inspection_lots (
    id BIGSERIAL PRIMARY KEY,                                       -- ロット内部ID（連番）
    lot_id VARCHAR(64) NOT NULL,                                    -- ロットID（製品ID等、例: 'MAT-10001'）
    section_code VARCHAR(32) NOT NULL,                              -- セクション内部コード（例: 'spring'）
    captured_at TIMESTAMP WITH TIME ZONE NOT NULL,                  -- 検査代表時刻（最初/最後のいずれか）
    UNIQUE (lot_id)                                                 -- ロットIDは一意
);

-- カメラショット（1ロット内で複数可、同一camera_idで複数行OK）
CREATE TABLE inspection_camera_shots (
    id BIGSERIAL PRIMARY KEY,                                                   -- ショット内部ID（連番）
    lot_id_ref BIGINT NOT NULL REFERENCES inspection_lots(id) ON DELETE CASCADE,-- ロット参照ID
    camera_id VARCHAR(64) NOT NULL,                                             -- カメラID（例: 'B-spring01', 'A-main01'）
    c4k_seq VARCHAR(32) NULL,                                                   -- 4K撮影シーケンスID（例: "A-4"｜JSONキー: "4k_seq"）
    fhd_seq VARCHAR(32) NULL,                                                   -- FHD撮影シーケンスID（例: "D-5"）
    status VARCHAR(16) NOT NULL CHECK (status IN ('PASS','FAIL','MISSING')),    -- 判定ステータス
    details TEXT NULL,                                                          -- 不良理由等（PASS時はNULL推奨）
    image_path TEXT NULL,                                                       -- 画像パス（MISSING時はNULL可）
    shot_seq INT NULL,                                                          -- 同一カメラ内の通番（1..n）※4K/FHD毎に任意
    captured_at TIMESTAMP WITH TIME ZONE NULL                                   -- ショット撮影時刻
);

備考: ロットの総合判定はビューやアプリ側で算出（同一camera_idで1つでもFAILがあればNG ルール）。4K/FHD 系列は DB 上では c4k_seq・fhd_seq で保持し、API レイヤでは 4k_seq・fhd_seq として返却します。必要ならマテリアライズドビューやトリガで日次集計を保持。

2. 検査サマリー取得

GET /api/inspections/summary 優先度: A

指定したセクション＋日付の検査サマリー（総数、良品/不良品数、不良原因）を取得します。ダッシュボードの統計表示で使用。

クエリパラメータ

パラメータ 説明 例
section 検査セクション（内部コード）。例: spring（バネ留め）, alayer（A層）。 ?section=alayer
date 対象の日付 (YYYY-MM-DD形式)。指定しない場合は本日。 ?date=2024-07-15

成功レスポンス (200 OK)

{
  "section": "alayer",
  "date": "2025-10-01",
  "total_count": 150,
  "pass_count": 145,
  "fail_count": 5,
  "pass_rate": 97,
  "fail_reasons": [
    { "reason": "位置ずれ", "count": 3 },
    { "reason": "糸ほつれ", "count": 1 },
    { "reason": "異物混入", "count": 1 }
  ]
}

3. 検査ロット取得

GET /api/inspections/lots 優先度: A

ロット単位で検査結果を取得。各ロットはセクション内の全カメラ結果（同一カメラの複数ショットを含み得る）を内包し、同一camera_idで1つでもFAILがあればロットNG とします。ページネーション対応。

クエリパラメータ

パラメータ 説明 例
section 検査セクション（内部コード）。例: spring（バネ留め）, alayer（A層）。 ?section=spring
date 対象の日付 (YYYY-MM-DD形式)。指定しない場合は本日。 ?date=2025-10-01
page ページ番号。 ?page=1
limit 1ページあたりの件数。 ?limit=50

{

    "current_page": 1,

3.1 ロット詳細（ショット一覧: 4K / FHD）A層検査用

ショット一覧は 撮影種別（4K/FHD）ごとにエンドポイントを分離 しました。UI では必要な種別のみリクエストし、各ショットの 4k_seq（DB列: c4k_seq）/ fhd_seq を用いて表示順や紐付けを制御します。

GET /api/inspections/lots/{lot_id}/shots/4K 任意

指定ロットの 4K ショットのみを返却します。4K シーケンス（4k_seq）単位で 重複を排除 し、同一 4k_seq を持つ行が複数あっても API では 1 件として返却します（最新ショット等、実装側で優先順位を定義）。

パスパラメータ

パラメータ 説明 例
lot_id 対象ロットID（パスで指定） /api/inspections/lots/MAT-10001/shots/4K

成功レスポンス例

{
    "lot_id": "MAT-10001",
    "shots": [
        {
            "camera_id": "B-spring01",
            "4k_seq": "A-4",
            "shot_seq": 2,
            "status": "PASS",
            "details": null,
            "image_path": "MAT-10001_B-spring01-4k-02.jpg"
        },
        {
            "camera_id": "B-spring01",
            "4k_seq": "A-5",
            "shot_seq": 1,
            "status": "FAIL",
            "details": "角度規定外",
            "image_path": "MAT-10001_B-spring01-4k-03.jpg"
        }
    ]
}
GET /api/inspections/lots/{lot_id}/shots/FHD/{4k_seq} 任意

指定ロット内で、特定の 4k_seq に紐づく FHD ショットのみ を返却します。4K 詳細ビューから FHD 連番を掘り下げる用途を想定しています。

パスパラメータ

パラメータ 説明 例
lot_id 対象ロットID（パスで指定） /api/inspections/lots/MAT-10001/shots/FHD/A-4
4k_seq 連動させたい 4K シーケンスID。該当する FHD ショットのみを抽出。 /api/inspections/lots/MAT-10001/shots/FHD/A-4

成功レスポンス例

{
  "lot_id": "MAT-10001",
  "4k_seq": "A-4",
  "shots": [
    {
      "camera_id": "B-spring02",
      "fhd_seq": "D-5",
      "shot_seq": null,
      "status": "PASS",
      "details": null,
      "image_path": "MAT-10001_B-spring02-fhd-01.jpg"
    },
    {
      "camera_id": "B-spring02",
      "fhd_seq": "D-5",
      "shot_seq": 2,
      "status": "FAIL",
      "details": "位置ずれ",
      "image_path": "MAT-10001_B-spring02-fhd-02.jpg"
    }
  ]
}

備考:

  • 4K シーケンスは DB 上は c4k_seq、API レスポンスでは 4k_seq として返します。FHD シーケンスは fhd_seq のままです。該当しないショットでは NULL を返して構いません。
  • 4K エンドポイントは 4k_seq ごとにユニークなショットのみ返却するため、クライアント側で重複除外を行う必要はありません。
  • shot_seq はカメラ＋種別の任意通番。旧 API 互換のため NULL 許容。
  • 4K/FHD を両方表示する UI は /shots/4K で候補を表示し、選択された 4k_seq を使って /shots/FHD/{4k_seq} を呼び出す作りにすると整合を保ちやすいです。

3.2 利用可能日付一覧

GET /api/inspections/sections/{section}/dates 任意

指定セクションのデータが存在する日付一覧（降順）を返します。

パスパラメータ

パラメータ 説明 例
section 検査セクション（内部コード） /api/inspections/sections/spring/dates

成功レスポンス

{
  "section": "spring",
  "dates": ["2025-10-02", "2025-10-01", "2025-09-30"]
}

4. 変更検知（シンプルGETポーリング）

GET /api/inspections/events/latest 優先度: A

直近の変更イベントの情報を返します。フロントは数秒おきにポーリングし、値が変化した時だけ /summary や /lots を再取得します。

レスポンス

{ "event": "lots.changed" }

フロント側の利用例

  • 間隔: 2〜5秒程度（環境と要件に合わせる）
  • 検知: 直近取得した値と比較し、変更ありのみデータAPIを再取得
  • 負荷対策: 連続変化時はデバウンス/スロットリングを実施
